## Bench setup & methodology

- **Fixture**: `vite-vanilla-ts-app` — generated by `scripts/bench/generate-vite-app-fixture.mjs`. ~86 lines of user TS in 4 source files (`main.ts`, `view.ts`, `metrics.ts`, `vite-env.d.ts`), plus `tsconfig.json` enabling `target: ES2020`, `lib: ES2020 + DOM + DOM.Iterable`, `moduleResolution: bundler`. Pulls in `vite/client.d.ts` ambient asset declarations from npm-installed `vite` package.
- **Bench harness**: `./scripts/bench/bench-vs-tsgo.sh --filter 'vite-vanilla-ts-app' --json`. Uses hyperfine for warmup + measured runs.
- **Profile harness**: `samply record --rate 4000 --unstable-presymbolicate` + `atos -o <binary> -l 0x100000000` for symbolication. The `release` profile keeps symbols (`strip = false`), `dist` strips them (`strip = "symbols"`), so profiling must use `release`.
- **Phase tracing**: `TSZ_PERF=1 ./tsz ...` enables the `wasm::perf=info` filter and emits one `INFO` event per phase with `phase=...` and `ms=...`. Zero overhead when not enabled.

## Phase breakdown of the vite case

From `TSZ_PERF=1 ./.target/debug/tsz --noEmit -p .../vite-vanilla-ts-live/tsconfig.json` (debug build, ~10× release-build wall-time but ratios are stable):

| Phase | Time (ms) | % of total |
|-------|----------:|-----------:|
| `read_sources` | 10 | 0.6% |
| `load_libs` | 100 | 6.2% |
| `build_program` | 61 | 3.7% |
| `build_lib_contexts` | 38 | 2.4% |
| **`collect_diagnostics`** | **1402** | **86.7%** |
| `emit_outputs` | 0 | 0% |
| **`compile_total`** | **1617** | — |

Release-mode equivalents (post-PRs): ~10ms / ~10ms / ~6ms / ~4ms / **~140ms** / 0ms / **~170ms** total.

**The 87% rule**: `collect_diagnostics` dominates. Every other phase combined is 13%. Optimization budget for this fixture belongs there — even a 50% improvement to `load_libs` saves only ~3% of total wall time.

## Leaf hot spots

Top leaf functions from a release-build samply profile post-#4433 + post-#4466 (PR #4513 hadn't landed yet at this snapshot):

| Family | Leaf samples (%) | Notes |
|---|---:|---|
| `hashbrown::HashMap::insert` + `reserve_rehash` + `hash_one` | ~250+ | Type interner ObjectShape inserts and hash-table growth |
| `tsz_solver::types::ObjectShape::hash` | ~100 | Per-property `Vec<PropertyInfo>` hash dominates |
| `Path::is_file` + `canonicalize` + `File::open_c` + `default_read_to_end` | ~140 | **Cached by PR #4513** — gave 15% on type-fest |
| `tsz_scanner::ScannerState::char_code_unchecked` + `is_identifier_part` | ~85 | Lib parsing scanner |
| `tsz_solver::visitors::walk_referenced_types` | ~35 | Allocates fresh `FxHashSet` + `Vec` per call |
| `tsz_common::comments::collect_comment_at` | ~30 | JSDoc / comment scanning |
| `rayon_core::sleep::Sleep::sleep` | ~150 | **Worker idle** — post-#4466 the lib pass saturates, but downstream work is single-threaded so workers sleep |
| `mi_*` (mimalloc) | ~60 | Allocator overhead from many small allocations |

The high `rayon::sleep` count is the most striking observation post-#4466: the workers wake briefly during the parallel lib pass and then idle through `collect_diagnostics`. This means **further parallelism inside checker phases is the second-biggest unrealized lever** after persistent state.

## PRs landed in this session

### PR #4433 — `perf(checker): precompile ambient module globs`

**What**: replaced `Vec<String>` of wildcard ambient module patterns with a pre-compiled `globset::GlobSet`. Four hot match call sites (`any_ambient_module_declared`, `wildcard_ambient_module_declared`, `matches_ambient_module_pattern`, the `module_exists_in_global_declared_modules` fallback) now do one `set.is_match(name)` call instead of recompiling every glob per match.

**Why hot**: vite's `vite/client.d.ts` declares ~30 wildcard ambient modules (`*.svg`, `*.css`, `*.module.css`, `*.png`, etc.). Every asset import was paying O(M) glob compiles per pattern.

**Win**: tsz 188 → 147ms `--quick` mode (factor 2.31× → 2.07× vs tsgo). Full-mode showed ~10-15% (more conservative). Reproducible across multiple runs.

**Test plan**: split `mod.rs` from 2045 → 1964 LOC by moving `GlobalDeclaredModules` into its own sibling file (architectural ceiling test was tripped by the inline definition). Added `project_env_tests::global_declared_modules_matches_vite_style_asset_patterns`.

### PR #4466 — `perf(core): parallelize checker lib reload`

**What**: `clone_lib_files_for_checker` re-parses + re-binds every lib file to build a fresh checker-side binder set (program binding mutates the binders during cross-lib symbol injection; checker can't reuse them without corrupting structural type relations like `Promise<T>` and `PromiseLike<T>`). The first lib pass (`load_lib_files_for_binding_strict`) was already parallel via rayon; the *second* pass was sequential. Switched to `maybe_parallel_iter!` to mirror it.

**Win**: structural symmetry fix; absolute saving on small fixtures is below noise floor on vite (~5-15ms range). Larger fixtures with full lib sets benefit more. **The headline value is parallel saturation of the second lib pass** — `rayon::sleep` would otherwise dominate that phase.

**Risk avoided**: rayon's order-preserving `collect` keeps the input order so `CheckerLibSet.files`'s positional consumers (`load_checker_libs`) are unaffected.

### PR #4513 — `perf(core): cache file-existence probes in module resolution`

**What**: thread-local `FILE_EXISTS: FxHashMap<PathBuf, bool>` cache wrapped by `cached_is_file()`. Used at 4 hot sites: `try_file_with_suffixes_and_extension`, `try_arbitrary_extension_declaration`, `resolve_explicit_unknown_extension`, and the `is_file()` in `file_probing.rs`. Exposes `clear_file_exists_cache()` for long-lived hosts (LSP, `--watch`).

**Win**: **15% on type-fest-project** (103.80ms → 88.45ms). Confirmed via parent-state A/B benchmarking. Vite shows ~3% (within noise — only 4 user files have too few imports to amortize cache pressure). rxjs unchanged (binding-heavy fast path).

**Lesson**: caches reward import locality. Workloads matching the access pattern see it; workloads without don't. The bench result shape (15% on monorepo-style fixtures, neutral on flat ones) is the *signature* of a correctly-targeted cache.

## Levers tried and rejected

| Lever | Predicted win | Actual on bench | Outcome |
|---|---|---|---|
| Pre-allocate `ConcurrentValueInterner::map` to `with_capacity_and_hasher(4096, ...)` | ~3ms (skip ~8 rehash chain) | Below ~15ms noise floor | Reverted; correctness fine but win unmeasurable |
| Dedup `canonicalize_or_owned(root_dir_path)` × 2 in `driver/core.rs` | ~0.5ms | Below noise | Not pursued; one-line refactor unworth the review burden |
| Lazy `ObjectShape::hash` (cache hash inside struct) | ~5ms (skip rehash hashing cost) | Not implemented — `mark_fresh_literal` mutates `flags` and would silently invalidate | Rejected on correctness complexity |
| `walk_referenced_types` thread-local `FxHashSet`/`Vec` reuse | ~5ms | Not implemented — callback can recurse into walk_referenced_types and alias the buffer | Rejected on aliasing risk |

## Why tsz is slower than tsgo on this fixture

**The structural answer**: tsz starts every invocation with empty caches and re-binds 47 lib files. tsgo maintains a long-lived checker singleton; lib types are interned once across all requests.

For an 86-line user file with ~50K lines of lib:
- **tsz**: parse + bind ~50K lib lines, intern ~8453 ObjectShapes, fill subtype+assignability caches from cold. ~170ms.
- **tsgo (warm)**: lib already interned, caches warm. ~80ms.

The **first-time-after-cold-start** cost is comparable for both (~170ms tsz, ~120ms tsgo per `--extendedDiagnostics` cold-cache projection). tsgo's win on benchmarks comes from the second-and-later runs in a long-lived process.

`--extendedDiagnostics` cache hit rates from the post-PR session:
- `Subtype cache: 30.8%`
- `Assignability cache: 25.9%`
- `Request cache: 0%` (5 misses, no hits)
- `Type interner: 8453 types, 2.4MB`

These are *cold start* numbers. In a long-lived process, hit rates approach 80%+.

## Future levers

Ordered by predicted impact on small-fixture wall time. The first two are the only ones that move the needle ≥ noise floor.

### 1. Persistent type-interner snapshot across invocations *(BIG)*

**Idea**: serialize the lib-derived ObjectShapes/FunctionShapes/TypeIds to a binary snapshot at build time (or on first run). On subsequent runs, mmap the snapshot and reconstitute the interner without parsing any lib files.

**Estimated savings**: 30-50ms on small fixtures (skips most of `load_libs` + `build_lib_contexts` + early `collect_diagnostics` lib type interning).

**Cost**: serialization format design, snapshot freshness invalidation when lib changes (TS submodule version), binary embedding tradeoffs (larger binary vs faster startup), back-compat across tsz versions.

**Roadmap scoping**: this is multi-PR. Recommend a dedicated workstream entry in `PERF_ARCHITECTURAL_PLAN.md` or its own doc.

### 2. Parallelize more inside `collect_diagnostics`

**Observation**: `rayon::sleep::Sleep::sleep` is ~4% of leaf time post-#4466. Workers idle through the checker phase because most checker work is per-file sequential within `parallel_check_files`.

**Idea**: identify checker sub-phases that are independent across files (declaration emit, JSDoc validation, ts7016 missing-types-package diagnostics) and parallelize them.

**Estimated savings**: 5-15ms depending on how much independent work exists.

**Risk**: shared state inside the checker (caches, interner) is safe via `&` (the interner is `Sync`); ensuring no inadvertent mutation in supposedly-read-only checker queries needs care.

### 3. ObjectShape::hash caching with explicit invalidation

**Idea**: add `cached_hash: AtomicU64` (0 = not computed) to `ObjectShape`. `mark_fresh_literal` and other mutators reset to 0. `Hash::hash` consults the cache first.

**Estimated savings**: ~5ms (skips rehash chain hash work). Most ObjectShapes are immutable after construction, so the invalidation surface is small.

**Cost**: every mutator must reset the cache. Compile-time-checked via a private `mutate(...)` method.

### 4. `walk_referenced_types` allocator reuse via stack-of-buffers

**Idea**: thread-local `Vec<(FxHashSet<TypeId>, Vec<TypeId>)>`. On entry, reuse the top buffer (pop, clear, then push when done). On recursion, push a fresh pair.

**Estimated savings**: ~5ms.

**Cost**: needs a careful guard pattern to handle panics; not worth the complexity unless other 5ms-tier wins are taken first.

### 5. `collect_comment_at` cache

**Idea**: cache `(NodeIndex, position) → Option<Atom>` for repeated JSDoc lookups.

**Estimated savings**: ~2-3ms.

**Cost**: small. Worth doing as a one-line PR if pursuing.

## Methodology notes

Lessons captured from this session that apply to any future tsz perf work.

### A/B against the *same* worktree

When a perf experiment shows surprising results, **stash the change, rebuild parent, run the same bench, compare**. Don't compare against a *different* checkout (e.g., a separate worktree or the canonical repo dir) — that introduces variables you don't control: target dir state, Cargo.lock, toolchain, FS layout.

This caught two false alarms in this session:
- **rxjs at 31ms** looked like my FS cache was breaking module resolution. Parent-state bench showed rxjs is just *a fast-binding fixture*; 31ms is the normal number.
- **zod TS18003 error** looked like my cache returned wrong false. Parent state errors identically — pre-existing fixture issue.

### `--quick` vs full bench mode

`--quick` mode runs 3 warmup + 5 measured hyperfine iterations. Full mode runs 3 warmup + 10+. On a 150ms benchmark, `--quick` has wider variance (~14ms / 9%) compared to full (~7ms / 4%). For PR-quality numbers, **always confirm with full mode** before quoting.

### macOS samply symbolication

The default samply output uses raw frame addresses. To get symbols on macOS:

```bash
# 1. Build with symbols (release profile keeps them; dist strips)
CARGO_TARGET_DIR=.target-bench cargo build --bin tsz --profile release

# 2. Profile with --unstable-presymbolicate (writes a sidecar .syms.json)
samply record --rate 4000 --unstable-presymbolicate --save-only \
  -o /tmp/profile.gz -- ./.target-bench/release/tsz <args>

# 3. Extract top leaf addresses and symbolicate via atos
python3 ... # parse profile.gz, extract top frame addresses
atos -o .target-bench/release/tsz -l 0x100000000 \
  $(addresses with 0x100000000 base added)
```

The `0x100000000` base is the standard arm64 macOS load address.

### `TSZ_PERF=1` is the right primary tool for phase analysis

Per-phase wall time via the `wasm::perf=info` filter is zero-overhead when disabled and gives the directional answer immediately:

```bash
TSZ_PERF=1 ./.target/debug/tsz --noEmit -p <project>/tsconfig.json 2>/tmp/trace.txt
```

Use this *before* samply. Identify the dominant phase first, then profile inside that phase.

### `--extendedDiagnostics` is gold

For any perf work, `--extendedDiagnostics` gives the canonical breakdown: I/O Read, Parse & Bind, Check, Emit, plus cache hit rates, type counts, memory totals. Read this **before** writing any code:

```bash
./.target-bench/dist/tsz --extendedDiagnostics --noEmit -p <project>/tsconfig.json
```

### When to stop iterating

A perf campaign has natural diminishing returns. Each clean PR in this session found a single hot spot with a contained fix (precompile globs, parallelize sequential pass, cache file-stats). The remaining hot spots either:

- Have correctness complications that require careful design (lazy hash with explicit invalidation, recursion-safe buffer reuse)
- Are architectural changes deserving their own roadmap entry (persistent type-interner snapshot)
- Are below the noise floor of the measurement harness

When the leaderboard transitions from "low-hanging" to "systemic," **stop adding speculative PRs and write up the findings** (this document). The next agent will benefit more from the analysis than from a marginal +2% PR.
