## Executive Summary

The repository has many repeated patterns. The largest remaining DRY payoff is from making a few high-churn concepts single-source:

1. Parse/bind/check/emit test harnesses: CLI, core WASM APIs, `tsz-wasm`, LSP tests, checker tests, emitter tests, and solver tests still rebuild nearly identical setup paths.
2. Project/file provider context: LSP project operations repeatedly fetch files, line maps, source text, binder state, arenas, and caches before calling feature providers.
3. Conformance execution/comparison residue: process-pool lifecycle, cache/test discovery, and script-side PASS/FAIL parsing are still duplicated across the runner and Python tooling.
4. AST traversal and source span handling: parser, binder, lowering, emitter, LSP, and diagnostics all have local traversal or span conversion logic that can drift.
5. WASM API residue: two exported surfaces still exist; compiler-option DTOs, diagnostic DTOs, and emit pipelines need consolidation on top of the enum parsing work already landed.
6. Remaining duplication within `tsz-checker`: primary-declaration selection (in flight), cross-file child checker lifecycle, heritage/member traversal, CommonJS export LHS classification, TS2322 rendering shapes, flow worklist enqueue (in flight).
